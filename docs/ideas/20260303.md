# Ideas — 2026-03-03

Brainstorm session on features and innovations beyond the core Crucible roadmap.

---

## Idea 1 — Temporal Debate Memory

Store `DebateTranscripts` keyed by `(file, symbol)`. When a finding is raised against a symbol that was previously reviewed, the agent checks: *"Was this exact pattern reviewed and accepted in session abc123?"* Agents develop institutional memory. The system learns what your team considers acceptable vs. not, without retraining any model.

Over time this creates a feedback loop: repeated false positives for a given pattern are suppressed; recurring real bugs are escalated. The memory store is queryable — `crucible memory show auth.rs::parse_token` reveals the full review history for a function.

**Unlocks:** team-specific tuning without fine-tuning; continuity across dev turnover.

---

## Idea 2 — Adversarial Patch Validation

After the Judge produces an `AutoFix`, spawn a second mini-council to *attack the fix itself*. Does the proposed patch introduce new issues? Does it actually solve the root cause or just mask the symptom? Does it break the invariants established in adjacent code?

The fix gets reviewed before it's offered to the developer. Recursive self-healing: Crucible doesn't just fix bugs, it validates its own fixes.

**Unlocks:** higher confidence auto-fix acceptance; eliminates the "the fix was worse than the bug" failure mode.

---

## Idea 3 — Predictive Pre-Staging

Run Crucible speculatively on `git add` (not just `git push`). By the time the developer types `git push`, the review is already done. Zero-latency gate — the hook just reads cached results from the daemon.

Combine with the daemon's warm connection pool: the developer gets sub-second feedback at push time regardless of how many agents are in the council.

**Unlocks:** eliminates the "waiting for review" friction; changes the psychological contract from "gate that slows me down" to "assistant that works ahead of me."

---

## Idea 4 — Semantic Regression Detection

Instead of only reviewing the diff, compute a semantic fingerprint of the changed functions (via tree-sitter + embedding). Compare against a fingerprint database of past buggy patterns — both from the org's own history and from public CVE datasets.

Flag: *"This function now resembles the CVE-2024-XXXX pattern from service Y, which was fixed in commit abc123."* Unlike static analysis rules, this catches novel variants of known bad patterns, not just exact matches.

**Unlocks:** CVE-aware review without maintaining a rule set; detects pattern recurrence across repos and teams.

---

## Idea 5 — Review Budget Negotiation

The developer declares a time/cost budget before pushing: *"I need a review in under 30 seconds, spend at most $0.05."* The coordinator dynamically selects which agents to invoke, how many rounds to run, and whether to use cheaper/faster models — optimizing review depth within the declared envelope.

Budget can be set globally in `.crucible.toml` or per-push as a flag. A "quick" preset uses one fast agent for one round; a "deep" preset uses all agents for three rounds.

**Unlocks:** Crucible adapts to urgency; removes the binary "full review or skip" tradeoff.

---

## Idea 6 — Confidence-Weighted Verdicts

Track each agent's historical accuracy per finding category (security, perf, architecture) by correlating their past findings with bugs that were later confirmed in production (via linked issue trackers, post-mortems, or manual labelling).

Over time, the verdict computation is weighted by *empirical agent accuracy*, not just role weight. An agent that has a 90% hit rate on security findings carries more weight than one at 60%, even if both have `role_weight = 2.0` in config.

**Unlocks:** self-calibrating council; the system gets better the longer it runs.

---

## Idea 7 — Mutation-Based Prompt Fuzzing

Periodically mutate the diff slightly — add/remove whitespace, rename a local variable, reorder struct fields — and check that the agents' findings are stable across semantically-equivalent mutations. If a Critical finding disappears on a trivially-equivalent mutation, flag it as a likely hallucination and downweight that finding.

Crucible stress-tests its own agents automatically, surfacing unreliable patterns in specific models or personas over time.

**Unlocks:** built-in hallucination detection; calibrated confidence scores without human labelling.

---

## Idea 8 — Cross-Repo Pattern Propagation

When a Critical finding is accepted and fixed in repo A, Crucible extracts the abstract pattern (via the Judge's explanation + the diff) and stores it in a shared pattern registry. All other repos in the Silverforge ecosystem periodically run a background scan against the registry.

One developer's accepted fix becomes a lint rule for the entire org. The pattern registry is versioned, diffable, and auditable — it's the org's accumulated code-review wisdom as a queryable artifact.

**Unlocks:** org-wide learning from individual reviews; no manual rule authorship required.

---

## Idea 9 — Debate Transcript as Training Data

Export anonymized `DebateTranscripts` in a format suitable for fine-tuning or preference learning (DPO/RLHF). The multi-round, adversarial debate naturally produces high-quality reasoning chains with:
- Initial position (round 1 analysis)
- Challenge (cross-pollination synthesis)
- Rebuttal and refinement (round 2+ debate)
- Final consensus (Judge summary)

This is exactly the kind of data that improves model performance on code review tasks. Crucible generates its own training signal as a byproduct of normal operation. An opt-in `crucible export --training-data` command packages transcripts with developer accept/reject labels attached.

**Unlocks:** continuous model improvement loop; Crucible becomes a data flywheel, not just a tool.

---

## Idea 10 — Speculative Branch Review

Rather than waiting for a push, run a background review of the *diff between the current working branch and main* whenever a significant file save occurs (debounced). Accumulate findings as the developer works.

By push time, Crucible has a rich picture of the entire branch's evolution — not just the final diff. The coordinator can surface: *"You introduced this vulnerability in commit 3 days ago; it survived 4 subsequent commits."* Context the final-diff-only view can never see.

**Unlocks:** longitudinal code intelligence; Crucible understands the journey, not just the destination.
